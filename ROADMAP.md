## P0 — 第 1–2 周：让用户能装上、试上、看见结果

### P0-1：修复打包后高精度模式失效

**问题描述**  
当前 `logo_eraser.spec` 中 `excludes = ['torch', 'iopaint', 'iopaint.*']`。打包出来的 `.app` / `.exe` 体积小（~80MB），但用户勾选"高精度模式"会直接报 `ModuleNotFoundError: torch`。这意味着 README 主打的卖点在最终产物中是坏的。

**根因**  
torch + iopaint 加进去后体积 1.5GB+，作者权衡过大小，但漏了"功能不可用 = 信任崩溃"这个产品代价。

**解决方案 A（推荐）：首次启动按需下载**
- 启动时检查 `~/.logo_eraser/runtime/torch_marker.json`
- 不存在 → 弹窗"启用高精度模式需下载 ~600MB 依赖，是否继续？"  
- 用户同意后下载并解压 torch wheel + MobileSAM 权重到 `~/.logo_eraser/runtime/`
- 把该目录加到 `sys.path`，标记完成

**解决方案 B：双安装包**
- `smart-logo-eraser-lite.dmg`（仅 padding 模式，80MB）
- `smart-logo-eraser-full.dmg`（含 PerSAM，1.5GB）
- 在下载页用对照表帮用户选

**子任务**
- [ ] 设计 runtime 目录结构和 marker 协议
- [ ] 在 server.py 启动逻辑加 `ensure_runtime()` 检查
- [ ] 写下载器（用 requests + 进度条 SSE 推送到前端）
- [ ] PyInstaller spec 加 `--onedir` + 测试在干净 Mac 上的安装体验
- [ ] 在 GitHub Release 上传两个产物（如选方案 B）

**验收**
- 全新 Mac（无 torch 安装）下载 .dmg → 装上 → 首次启动 → 同意下载 → 能正常用高精度模式擦除 demo 图
- 不勾"高精度模式"时无需下载，padding 模式立即可用

**预计工作量**：8–12 小时

### P0-2：内置 examples/ 示例素材

**问题描述**  
用户 clone 仓库后没有可立即试跑的图。第一次使用门槛 = 自己找带水印的图 = 90% 用户在这一步流失。

**方案**
- 在仓库根目录加 `examples/` 目录
- 准备两组场景，每组含「样图 × 1 + 批量图 × 3」
  - 场景 1：电商产品图 + STOCK4U 水印
  - 场景 2：摄影作品 + PHOTOMARK 水印
- 用本工具生成水印的脚本放 `scripts/make_examples.py`（可复现）
- README 加 "5 秒上手" 小节，引用 examples

**注意**  
所有示例图用 jimu_image_generation 合成生成或公开 CC0 素材，避免侵权风险。

**子任务**
- [ ] 写 `scripts/make_examples.py`：合成水印 + 输出干净版本作为参考标签
- [ ] 生成 2 组 × 4 张 = 8 张样图，提交进 `examples/`
- [ ] `README.md` 加 "5 秒上手" 章节，含一行命令演示
- [ ] 桌面应用 UI 增加"试试示例素材"按钮，一键加载

**验收**
- 新用户 clone 后照着 README 走，3 分钟内能跑出第一张擦除结果

**预计工作量**：4–6 小时

### P0-3：录制 30s Demo GIF / MP4

**问题描述**

**方案**

- 输出尺寸 800px 宽，< 5MB（GitHub README 渲染流畅）

**子任务**

- [ ] 录 1 遍粗剪 → 看节奏 → 调整

**验收**
- GitHub README 顶部 GIF 自动播放，不让人觉得卡顿

### P0-4：单图速擦入口

**问题描述**  
用户只有 1 张图要擦时，仍然要走完整流程：建模板 → 上传批量 → 处理。这条路径对"首次使用"过于重型。

**方案**  
在主界面顶部加一个 Tab 切换：「单图速擦 / 批量擦除」。
- 单图模式：拖入一张图 → Canvas 上直接框 logo → 点擦除 → 立即出结果
- 不保存模板、不上传批量、最少点击数
- 内部仍然用 PerSAM-F（参考特征就是这张图本身的框选区域）

**子任务**
- [ ] HTML：加 Tab 切换组件，复用现有 Canvas
- [ ] 新增 `/api/single-erase` 接口
- [ ] server.py 实现"框 → PerSAM → 擦除"单图流程（参考特征不缓存到 templates.json）
- [ ] 前端 UI：上传 / 框选 / 擦除按钮 / 结果对比

**验收**
- 新用户安装应用，不读 README 不看引导，3 步内擦出第一张图

**预计工作量**：6–8 小时

### P0-5：失败兜底（手动修复入口）

**问题描述**  
某张图模板匹配失败 → 当前结果直接标 `no_match` 扔掉；用户拿到一堆"未处理"图却无路可走。这是用户挫败感的最大来源。

**方案**
- 结果列表中 `no_match` 项前面加"手动定位"按钮
- 点击 → 打开该图的 Canvas → 让用户手动框 logo 区域
- 框完点"用此区域擦除" → 走 SAM + 擦除流程 → 替换结果

**子任务**
- [ ] 前端：结果项分两个状态 (ok / pending-manual)，pending 项可点击进入修复模式
- [ ] 新增 `/api/manual-erase` 接口（仅当前用户框的单 bbox + 当前图）
- [ ] 修复流程复用 PerSAM 引擎，参考特征用模板库中第一个匹配的模板（或当前 bbox 自身）
- [ ] 完成后结果合并回主网格

**验收**
- 一批 10 张图，5 张匹配失败 → 用户依次点失败项手动框 → 全部能擦干净
- 用户不需要离开桌面应用、不需要去 PS 补救

**预计工作量**：6–10 小时

## P1 — 第 3–6 周：核心交互闭环 + 国际化基础

### P1-1：mask 微调画笔

**问题描述**  
SAM 分割结果差一两个像素时，目前用户没有补救手段。商业竞品（Cleanup.pictures、Photopea）的核心交互就是画笔。

**方案**
- 擦除结果项点击放大后进入"微调"模式
- 提供两个工具：「+」涂笔（加进 mask）、「−」涂笔（从 mask 移除）
- 实时预览：每次涂笔后立即重新走擦除（fill 模式 < 100ms，体感即时）
- 调整完保存 → 覆盖结果

**子任务**
- [ ] 设计 mask 编辑数据模型（base mask + delta strokes）
- [ ] 前端：增加 mask 编辑 Canvas 层（独立于原图层）
- [ ] 新增 `/api/refine-mask` 接口：接收 base mask + strokes → 返回新结果
- [ ] 性能：refine 必须 < 200ms 才能让画笔流畅
- [ ] UI：画笔大小、撤销 / 重做、确认 / 取消

**验收**
- 用户能在 30 秒内修好一个 mask 错误
- 改完直接保存覆盖，不破坏其他结果

**预计工作量**：12–16 小时（最大一项）

---

### P1-2：进度反馈 / per-file 状态

**问题描述**  
批量 50 张图处理时，前端只有 indeterminate 进度条。用户不知道处理到第几张、卡哪了、还要多久。

**方案**
- server 改用 SSE 或 WebSocket 推送 per-file 状态：start → matched → segmenting → erasing → done
- 前端结果网格"占位卡片"先渲染，每张图按事件更新状态
- 估算剩余时间（用前 3 张的平均耗时外推）

**子任务**
- [ ] server.py 把 process_batch 改成 generator，每张图 yield 一个事件
- [ ] 增加 `/api/process-stream`（SSE），保留旧 `/api/process` 用于兼容
- [ ] 前端 EventSource 接入，更新结果项实时状态
- [ ] 加估算剩余时间 UI

**验收**
- 50 张图批量处理时，每张完成都能立即看到结果（不用等整批跑完）

**预计工作量**：6–8 小时

---

### P1-3：结果 mask 叠加预览

**问题描述**  
擦除结果只能看到最终图，看不到"AI 在哪里擦的"。用户判断 AI 是否做对了的成本高。

**方案**
- 每个结果项加一个"显示 mask"开关
- 开关打开 → 在结果图上叠加半透明红色覆盖（show mask 区域）
- 服务端处理时把 mask 一并保存（`<batch_id>/<name>_mask.png`），前端按需加载

**子任务**
- [ ] server.py：擦除时保存 mask PNG 到 outputs 目录
- [ ] 结果 API 返回 mask URL
- [ ] 前端：结果项加 toggle，开关切换叠加渲染

**验收**
- 用户能在结果上看到 AI 实际处理区域，与原图对比

**预计工作量**：3–4 小时

### P1-5：拖拽 / 粘贴 / URL 输入

**问题描述**  
当前唯一上传路径是点按钮选文件。在 Mac/Windows 桌面应用里这是非常基础但常被忽视的体验项。

**方案**
- 整个 Canvas 区域接受拖拽（dragover / drop 事件）
- Cmd+V / Ctrl+V 直接处理剪贴板图片
- 加一个"贴 URL"输入框，server 端代理下载

**子任务**
- [ ] 前端 Canvas 绑定 drag/drop 处理
- [ ] 全局 paste 监听，转 file 上传
- [ ] 新 API `/api/upload-by-url`：服务端下载后转入正常 upload 流程
- [ ] 大小 / 类型校验

**验收**
- 用户从浏览器拖图到应用、Cmd+V 截图、贴 URL 三种方式都能正常处理

**预计工作量**：3–5 小时

### P1-6：模板库管理增强

**问题描述**  
当前模板库只支持「列表 + 重命名 + 删除」。重度用户（一个商家有几十个 logo 需要管理）会感到吃力。

**方案**
- 模板加 `group` 字段，前端按分组折叠展示
- 增加"导出模板包"功能：模板 metadata + 样图 + 裁剪图打包成 zip
- 增加"导入模板包"，可在不同电脑间迁移
- 模板缩略图支持放大预览

**子任务**
- [ ] templates.json 增加 group 字段（向后兼容）
- [ ] 前端：分组折叠 UI，拖拽换组
- [ ] `/api/export-templates`：zip 打包
- [ ] `/api/import-templates`：zip 解包 + 合并
- [ ] 缩略图点击放大 lightbox

**验收**
- 模板数 > 20 时仍能高效定位和管理
- 换电脑通过 zip 一键迁移整个模板库

**预计工作量**：6–8 小时

## P2 — 第 7-12 周：延展场景 / 进阶能力（择优）

下面这些不细化为子任务，先列方向，做完 P0/P1 再回头评估优先级。

### 视频水印移除
按帧抽样 + SAM 跟踪 + ffmpeg 合成。技术难度中等，但用户基数 10×。延伸到 TikTok / 油管搬运党人群。

### 旋转 / 弯曲水印支持
模板匹配阶段加入旋转搜索（每张图 4-8 个角度），或换 LoFTR / RoMa feature matching。半透明角章场景能覆盖。

### 自动水印检测
集成 HuggingFace 上现成的 watermark 检测模型（如 fancyfeast/joycaption-watermark-detection），用户**不框也能跑**。把工具从"会用的人才用"升级为"小白也能用"。

### 自托管服务模式
把桌面应用拆出一个纯 Web 版（Flask + Docker），配合多用户 + API key。适合内部团队部署。

### 命令行批处理增强
当前 `scripts/batch_remove_logo.py` 是早期版本，没接入 PerSAM。可以重写为支持 PerSAM 模式的纯 CLI，方便 CI / 服务器场景。

## 衡量指标（每个里程碑后自检）

**首次使用成功率**  
新用户从下载 → 看到第一张擦除结果的转化率。目标 P0 后 > 80%。

**单次任务完成率**  
用户开一个会话能跑完一整批图的比例。目标 P1 后 > 90%（含失败兜底后能完成）。

**用户从挫败中恢复率**  
SAM / 模板匹配出问题时，用户是放弃还是修补。目标 P1 后 > 70%。

**GitHub Star / Fork 增长**  
做完 P0-3 GIF + P1-4 双语后跟踪 7 天增长曲线。
